# engineer-assignment-mdp

単一技術者の案件アサイン問題を、有限状態 MDP として最小構成で再現する研究用プロトタイプです。TeX ノート `engineer_assignment_mdp_note.tex` の考え方をベースにしつつ、**実装はユーザー依頼文の明示仕様を優先**しています。

## 1. 問題設定の整理

### 問題設定
待機中の 1 人の技術者に対し、案件担当マネージャーが各期に
- 高単価案件を狙う
- 低単価案件を狙う
- 研修を受ける
- 何もしない

のいずれかを選び、期待割引累積報酬を最大化します。

### 状態
状態は `s = (z, k)` とし、
- `z ∈ {B, H, L}`
  - `B`: ベンチ
  - `H`: 高単価案件に稼働中
  - `L`: 低単価案件に稼働中
- `k ∈ {0, 1, ..., K}`

です。既定値では `K=3` なので状態数は 12 です。

### 行動
行動は `A = {a_H, a_L, a_T, a_N}` とし、
- `a_H`: 高単価案件を狙う
- `a_L`: 低単価案件を狙う
- `a_T`: 研修を受ける
- `a_N`: 何もしない / 現状維持

です。

### 合法行動
- `z=B` のとき: `{a_H, a_L, a_T, a_N}`
- `z=H` または `z=L` のとき: `{a_N}`

### 遷移
依頼文どおり、ベンチでは応募成功・研修成功に応じて遷移し、稼働中は継続または離脱のみを扱います。

- `B` で `a_H`: `p_H(k)` で `(H, k)`、`1-p_H(k)` で `(B, k)`
- `B` で `a_L`: `p_L(k)` で `(L, k)`、`1-p_L(k)` で `(B, k)`
- `B` で `a_T`: `q_T(k)` で `(B, min(k+1, K))`、失敗時は `(B, k)`
- `B` で `a_N`: 必ず `(B, k)`
- `H` で `a_N`: `d_H` で `(H, k)`、`1-d_H` で `(B, max(k-1, 0))`
- `L` で `a_N`: `d_L` で `(L, k)`、`1-d_L` で `(B, max(k-1, 0))`

### 報酬
依頼文の指定に合わせて、**遷移後状態 `z'` に基づく報酬**として実装しています。

- `z' = H` のとき `+R_H`
- `z' = L` のとき `+R_L`
- `z' = B` のとき `-C_w`
- さらに `a_T` のとき追加で `-C_edu`

### 目的関数
各方策 `π` について、

`E[sum_t gamma^t r_t]`

を最大化します。

### ベースライン方策
- `baseline_1_high_only`: 常に高単価案件狙い
- `baseline_2_low_only`: 常に低単価案件狙い
- `baseline_3_train_then_high`: `k<=1` で研修、`k>=2` で高単価
- `baseline_4_myopic_utilization`: 各 `k` で `max(p_H(k), p_L(k))` を与える行動

### 評価指標
- 平均累積報酬
- 平均稼働率
- H 滞在割合
- L 滞在割合
- B 滞在割合
- 平均適合度 `k`
- 平均ベンチ脱出時間

## 2. ファイル構成と役割

- `mdp_env.py`: 状態・行動・遷移・報酬・パラメータの定義
- `solvers.py`: 価値反復法と方策反復法
- `baselines.py`: ベースライン方策群
- `simulate.py`: シミュレーション、比較、可視化
- `notebook.ipynb`: Colab でそのまま試せる notebook

## 3. TeX ノートと実装の対応

実装の中心は `EngineerAssignmentMDP` クラスです。

- 状態空間の列挙: `mdp_env.py` の `self.states`
- 行動空間の列挙: `mdp_env.py` の `self.actions`
- 合法行動制約: `self.legal_actions_map`
- 遷移確率 `P(s'|s,a)`: `self.transition_probabilities`
- 報酬 `R(s,a,s')`: `self.rewards`
- 確率和の検証: `validate_transition_probabilities()`
- 価値反復法: `solvers.value_iteration()`
- 方策反復法: `solvers.policy_iteration()`
- ベースライン: `baselines.py`
- 評価と可視化: `simulate.py`

## 4. 実行方法

### ローカル実行
```bash
python simulate.py
```

これで
1. 遷移確率の検証
2. 各方策の比較結果出力
3. `policy_comparison.png` の保存

を行います。

### Colab 実行
`notebook.ipynb` を Google Colab にアップロードして、上から順に実行してください。追加ライブラリは `numpy` と `matplotlib` のみです。

## 5. 実装上の明記事項

TeX ノートには、途中で以下の別案も記載されています。
- 報酬を「遷移前状態ベース」で定義する案
- 稼働継続中に確率 `δ` で適合度が低下する案

今回はユーザー依頼文に合わせて、以下を採用しました。
- 報酬は **遷移後状態ベース**
- 稼働中の適合度低下は **離脱時にのみ 1 低下**

研究ノートとの差分であるため、将来比較しやすいよう README に明記しています。

## 6. 将来拡張ポイント

### K の一般化
すでに `MDPParameters.K` と `p_H / p_L / q_T` の長さチェックを入れているため、`K` と配列長を合わせれば状態空間は自動で一般化できます。

### `d_H`, `d_L` の k 依存化
現在は定数ですが、`d_H[k]`, `d_L[k]` の配列に変更し、`_build_transition_and_reward_tables()` 内で `k` に応じて継続確率を参照すれば拡張できます。

### 報酬を案件粗利ベースに変更
`reward_from_next_state()` を差し替えれば対応できます。たとえば `R_H`, `R_L` を売上ではなく粗利にし、待機コストや教育コストの内訳を追加する形にできます。

### 複数技術者への拡張
最も素直なのは、単一技術者 MDP をベースに
- 独立な複数エージェントのシミュレーションに拡張する
- もしくは全技術者の直積状態を持つ MDP / 近似動的計画に拡張する

という二段階です。研究プロトタイプとしては、まずは単一技術者環境を複数サンプル並列に回す評価系を作るのが実務的です。

### 感度分析
`MDPParameters` を複数パターン走査し、各パラメータセットに対して
- 最適方策
- 平均累積報酬
- 稼働率

を再計算するループを追加すれば実装できます。特に `p_H`, `p_L`, `q_T`, `d_H`, `d_L`, `C_edu` を動かすと、研修の価値や高単価狙いの閾値がどのように変わるかを分析できます。
